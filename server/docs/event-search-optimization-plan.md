## 2. 方案对比矩阵

### 方案 1: pg_trgm + GIN 索引（PG 内部）

| 项目 | 说明 |
|---|---|
| **原理** | 将字符串拆成 3 字符片段（trigram），建 GIN 倒排索引，`LIKE '%keyword%'` 自动走索引 |
| **优点** | 零外部依赖；对现有代码改动最小；严格子串匹配语义 |
| **缺点** | 索引体积大（约为原数据 1-3x）；写入时有索引维护开销；keyword < 3 字符时退化为全扫描 |
| **资源开销** | 中等（额外磁盘 + 写入 CPU） |
| **适合** | 前端交互搜索，中低频查询 |
| **不适合** | 规则引擎大量并发子串匹配（GIN 索引在高频查询下锁竞争） |

### 方案 2: 按时间分区表（PG 内部）

| 项目 | 说明 |
|---|---|
| **原理** | 按天/小时对表做 Range 分区，查询时通过 `event_time` 条件自动裁剪分区 |
| **优点** | 查 24 小时数据时只扫描 1-2 个分区；数据过期直接 `DROP PARTITION`，比 DELETE 快 1000x；与其他方案可叠加 |
| **缺点** | 单独使用不解决 LIKE 问题，必须配合其他索引方案 |
| **资源开销** | 低（几乎无额外开销） |
| **适合** | **所有场景的基础设施**，无论选哪个方案都应该做 |

### 方案 3: 写入时流式匹配（规则引擎专用）

| 项目 | 说明 |
|---|---|
| **原理** | 事件写入 DB 前，在 Go 服务层用 Aho-Corasick / 正则引擎对规则进行流式匹配 |
| **优点** | 零查询开销（不查 DB）；实时性最高（事件到达即匹配）；规则数量 100-1000 条时 AC 自动机性能极好（O(n) 复杂度） |
| **缺点** | 规则更新需要重建 AC 自动机（半动态可接受）；回溯历史数据需要另外的方案 |
| **资源开销** | 极低（纯内存计算） |
| **适合** | **规则引擎实时匹配**（IOC 域名、恶意路径、可疑命令） |
| **不适合** | 前端交互式搜索（用户输入的关键词不可预知） |

### 方案 4: Elasticsearch 双写

| 项目 | 说明 |
|---|---|
| **原理** | 事件写入 PG 的同时异步写入 ES，搜索请求走 ES |
| **优点** | 搜索能力最强（支持子串、分词、聚合分析）；与 PG 职责分离 |
| **缺点** | **资源开销大**（ES 内存需求 >= 4GB JVM heap）；多一套组件运维；私有化部署增加复杂度 |
| **资源开销** | 高（内存 8GB+，磁盘与 PG 同量级） |
| **适合** | 数据量达到亿级、需要复杂聚合分析时 |
| **不适合** | 当前阶段（资源有限 + 私有化部署） |

### 方案 5: ClickHouse 替代方案

| 项目 | 说明 |
|---|---|
| **原理** | 将事件数据写入 ClickHouse 列式存储，利用其暴力扫描能��做 LIKE 查询 |
| **优点** | 列式存储天然压缩好（10x+）；全扫描速度远超 PG（GB/s 级）；tokenbf_v1 索引支持子串查询加速 |
| **缺点** | 引入新数据库；不适合 OLTP 场景（PG 仍需保留做主库）；运维学习成本 |
| **资源开销** | 中等（内存 2-4GB 即可起步） |
| **适合** | 日增数据过亿 + 需要分析能力时 |
| **不适合** | 当前数据规模下 ROI 不高 |

## 3. 推荐实施路线

结合约束条件（**私有化部署、资源有限、规则引擎优先、严格子串匹配、低并发**），推荐分三阶段渐进实施：

### 阶段一（基础）：分区表 + 写入时流式匹配

**目标：解决规则引擎实时匹配（最高优先级）**

```
事件流入 → Go 服务层 Aho-Corasick 匹配 → 命中则生成告警
                ↓
         写入 PG 分区表（按天分区）
```

核心工作：

1. 四张表全部改为按天分区（基于 `event_time`）
2. Go 层引入 Aho-Corasick 库（如 `cloudflare/ahocorasick`），100-1000 条规则匹配耗时微秒级
3. 规则更新时重建自动机（半动态场景，秒级完成）
4. 过期分区直接 DROP，解决数据清理问题

**收益**：规则引擎零 DB 查询开销；分区表让所有现有查询缩小扫描范围。

### 阶段二（增强）：pg_trgm + GIN 索引

**目标：解决前端交互式搜索**

核心工作：

1. 在分区表基础上，对 `exe_path`, `args`, `domain`, `new_path`, `comm` 等字段建 pg_trgm GIN 索引
2. 由于分区 + 时间过滤，GIN 索引只需覆盖当天数据，体积可控
3. `LIKE '%keyword%'` 可以自动走索引

**收益**：前端搜索从全表扫描变为索引查找，查询延迟降到毫秒级。

### 阶段三（可选）：引入 ClickHouse / ES

**触发条件：当日增超过 1 亿行，或需要复杂聚合分析时**

- 如果主要是搜索 + 简单聚合 → ClickHouse（资源更省）
- 如果需要复杂全文搜索 + 可视化 → ES + Kibana
