# 任务执行顺序和依赖关系

## 📋 依赖关系分析

### 🔗 关键依赖链

```
配置模块 (任务3) 
  ↓
  ├─→ transport 模块 (任务5) - 需要读取 Server 地址
  └─→ main.go (任务9.2) - 需要初始化配置

proto 模块 (任务4)
  ↓
  └─→ transport 模块 (任务5) - 需要消息定义

agent 模块 (任务1)
  ↓
  ├─→ transport 模块 (任务5.3) - 需要 Agent ID
  └─→ heartbeat 模块 (任务7.1) - 需要 Agent ID 和状态

host 模块 (任务2)
  ↓
  ├─→ transport 模块 (任务5.3) - 需要主机信息
  └─→ heartbeat 模块 (任务7.1) - 需要主机信息

buffer 模块 (任务8)
  ↓
  ├─→ transport 模块 (任务5.3) - 需要写入数据
  └─→ heartbeat 模块 (任务7) - 需要写入状态

resource 模块 (任务6)
  ↓
  └─→ heartbeat 模块 (任务7.2-7.4) - 需要资源监控

heartbeat 模块 (任务7)
  ↓
  └─→ main.go (任务9.4) - 需要启动心跳守护进程

transport 模块 (任务5)
  ↓
  └─→ main.go (任务9.4) - 需要启动传输守护进程
```

---

## 🚀 推荐执行顺序（按阶段）

### 📍 阶段一：基础设施（可并行开发）

#### 批次 1.1：独立基础模块（可同时开始）
```
任务1.1: Agent ID 生成 - DMI 和 MAC 地址
任务1.2: Agent ID 生成 - machine-id 回退
任务2.1: 主机名获取
任务2.2: IPv4 地址收集
任务2.3: 平台信息收集
任务3.1: 定义配置结构体
```

#### 批次 1.2：依赖批次1.1
```
任务1.3: Agent ID 持久化 (依赖 1.1, 1.2)
任务2.4: host 包初始化 (依赖 2.1, 2.2, 2.3)
任务3.2: 配置文件读取 (依赖 3.1)
```

#### 批次 1.3：完成阶段一
```
任务1.4: Agent 包初始化 (依赖 1.1, 1.2, 1.3)
任务1.5: 状态管理 (独立，但建议在 1.4 之后)
任务3.3: 配置验证和默认值 (依赖 3.2)
任务3.4: 全局配置访问 (依赖 3.3)
```

**阶段一完成检查点**：✅ Agent ID 生成、主机信息收集、配置管理都已完成

---

### 📍 阶段二：通信层（依赖阶段一）

#### 批次 2.1：proto 定义（可独立开始，但建议在阶段一完成后）
```
任务4.1: 定义基础消息类型
任务4.2: 定义 PackagedData 消息 (依赖 4.1)
任务4.3: 定义 Command 消息 (依赖 4.1)
任务4.4: 定义 Transfer 服务 (依赖 4.1, 4.2, 4.3)
```

#### 批次 2.2：proto 代码生成
```
任务4.5: 生成 Go 代码 (依赖 4.1-4.4)
任务4.6: 验证 proto 定义完整性 (依赖 4.5)
```

#### 批次 2.3：transport 基础（依赖阶段一和 proto）
```
任务5.1: 基础连接管理 (依赖 任务3, 任务4)
任务5.2: 连接重试机制 (依赖 5.1)
任务5.6: transport 包初始化 (独立，但建议在 5.1 之后)
```

#### 批次 2.4：transport 数据传输（依赖 buffer）
```
任务5.3: 数据发送逻辑 (依赖 任务1, 任务2, 任务4, 任务8)
任务5.4: 命令接收逻辑 (依赖 任务4)
```

#### 批次 2.5：transport 完成
```
任务5.5: 传输守护进程 (依赖 5.1-5.4)
```

**阶段二完成检查点**：✅ proto 定义完成，transport 可以建立连接并传输数据

---

### 📍 阶段三：状态上报（依赖阶段一和阶段二部分）

#### 批次 3.1：resource 模块（可独立开始）
```
任务6.1: 进程资源监控基础功能
任务6.2: IO 速度计算 (依赖 6.1)
任务6.3: 系统资源获取 (独立)
任务6.4: Linux 特定功能 (独立)
任务6.5: resource 包初始化 (依赖 6.1-6.4)
```

#### 批次 3.2：heartbeat 模块（依赖 resource 和阶段一）
```
任务7.1: Agent 状态收集 - 基础信息 (依赖 任务1, 任务2)
任务7.2: Agent 状态收集 - 性能指标 (依赖 任务6)
任务7.3: Agent 状态收集 - 网络和负载 (依赖 任务6)
任务7.4: Plugin 状态收集 (依赖 任务6)
任务7.5: 心跳守护进程 (依赖 7.1-7.4, 任务8)
```

**阶段三完成检查点**：✅ 可以收集并上报 Agent 和 Plugin 状态

---

### 📍 阶段四：集成（依赖前面所有阶段）

#### 批次 4.1：buffer 模块（可提前开始，建议在阶段二之前）
```
任务8.1: WriteRecord 方法
任务8.2: 对象池优化 (独立)
任务8.3: 优化 buffer 写入逻辑 (依赖 8.1)
```

#### 批次 4.2：main.go 集成（依赖所有模块）
```
任务9.1: logger 初始化 (独立，建议最先完成)
任务9.2: 配置初始化 (依赖 任务3)
任务9.3: pid 文件锁 (独立)
任务9.4: 守护进程启动 (依赖 任务5, 任务7)
任务9.5: 信号处理和清理 (依赖 9.4)
```

**阶段四完成检查点**：✅ Agent 可以完整启动并运行

## 🎯 关键路径（Critical Path）

**最长依赖链**（必须按顺序执行）：
```
3.1 配置结构体 
  → 3.2 配置文件读取 
  → 3.3 配置验证 
  → 3.4 全局配置访问 
  → 5.1 基础连接管理 
  → 5.2 连接重试机制 
  → 5.3 数据发送逻辑

  → 5.5 传输守护进程 
  → 9.4 守护进程启动
```

**总时长**：约 13.5 小时（关键路径）

## 💡 并行开发建议

### 方案一：4人并行开发（推荐）

**开发人员 A（基础设施）**：
- 批次 1.1 → 批次 1.2 → 批次 1.3
- 完成后可协助其他模块

**开发人员 B（通信层）**：
- 批次 2.1 → 批次 2.2 → 批次 2.3 → 批次 2.4 → 批次 2.5
- 需要等待：批次 1.3（配置模块）、批次 2.2（proto 代码生成）

**开发人员 C（状态上报）**：
- 批次 3.1 → 批次 3.2
- 需要等待：批次 1.3（agent、host 模块）

**开发人员 D（集成和 buffer）**：
- 批次 4.1（buffer）→ 批次 4.2（main.go）
- buffer 可提前开始，main.go 需要等待所有模块

### 方案二：快速迭代（2人）

**第一周**：
- 人员1：阶段一（基础设施）
- 人员2：proto 模块 + buffer 模块

**第二周**：
- 人员1：transport 模块（等待 proto 和配置）
- 人员2：resource 模块 + heartbeat 模块

**第三周**：
- 人员1：main.go 集成
- 人员2：测试和优化

---

## ⚠️ 注意事项

1. **proto 模块必须最先完成**：其他模块都依赖它
2. **配置模块优先级高**：transport 和 main.go 都依赖它
3. **buffer 模块可提前开发**：不依赖其他模块，建议在阶段二之前完成
4. **测试时机**：
   - 每个批次完成后进行单元测试
   - 阶段完成后进行集成测试
   - 最终进行端到端测试

5. **阻塞点**：
   - 任务 5.3（数据发送）需要等待 buffer 模块
   - 任务 7.1-7.4（heartbeat）需要等待 resource 模块
   - 任务 9.4（守护进程启动）需要等待 transport 和 heartbeat
